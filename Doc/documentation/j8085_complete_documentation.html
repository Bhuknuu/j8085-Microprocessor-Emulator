1. Project Overview and Architectural Thesis

The j8085 Microprocessor Simulator is a software reconstruction of the Intel 8085A microprocessor, a chip that Intel released in 1976 as the successor to the 8080. The physical 8085 contains approximately 6,500 transistors arranged on a single piece of silicon, providing an 8-bit data bus and a 16-bit address bus. It can address up to 64 kilobytes of memory. Our simulator reproduces the observable behaviour of this chip entirely in Java, meaning that any assembly language programme written for the real 8085 will produce identical register states, flag states, and memory contents when executed in our software model.

The fundamental question that governs every design decision in this project is: how do you translate a physical, parallel, electrically-driven machine into a sequential, software-driven model without losing behavioural fidelity? The answer lies in composition. Rather than building one monolithic class that handles everything from flag computation to memory addressing, we decompose the processor into the same logical units that an electrical engineer would recognise on the chip's block diagram.

The project consists of nine Java source files. Each file corresponds to one identifiable subsystem of the hardware:

Today, Day 1, we examine the four foundational files that everything else is built upon. Without the error system, the simulator cannot communicate failures. Without the Memory interface, the CPU has no contract for how it reads and writes bytes. Without the ALU, the CPU cannot compute anything. Without the InstrucSet interface, there is no formal specification of what instructions exist. These four files form the bedrock.

2. SimulatorException: The Error Backbone

SimulatorException.java SimulatorException.java 51 lines

Every simulator must handle errors, but the nature of the errors in a microprocessor simulator is fundamentally different from ordinary software errors. When a user writes MOV A, X (where X is not a valid register name), the error is not a null pointer or an array index violation in the Java sense. It is a hardware-level semantic error: the instruction references a register that does not exist on the physical chip. Our exception system must capture this distinction. It must tell the user not just "something went wrong" but "this specific kind of hardware-level violation occurred, and here is the memory address where it happened."

The ErrorType Enum

public class SimulatorException extends Exception {
    public enum ErrorType {
        InvalidMemoryAddress,     // Address outside 0x0000-0xFFFF
        InvalidRegister,          // Unrecognised register name
        InvalidOpcode,            // Unrecognised instruction opcode
        InvalidData,              // Data value outside valid range
        StackOverflow,            // SP went below 0x0000
        StackUnderflow,           // SP went above initial value
        MemoryOutOfBound,         // Access outside allocated range
        HaltEncounter,            // HLT executed (not truly an error)
        SyntaxError,              // Assembly syntax invalid
        UndefinedAddress          // Label doesn't exist
    }
What is happening here. We define a Java enum called ErrorType nested inside the exception class. An enum is a special type in Java that represents a fixed set of named constants. Rather than throwing a generic Exception with only a string message, every SimulatorException carries a typed constant that tells the caller precisely which category of failure occurred. The caller can then programmatically react to the error type (for example, displaying a different help message for a syntax error versus a memory violation) without having to parse the human-readable message string.
Why an enum and not integer error codes? Integer error codes (like ERROR_CODE = 3) are fragile. A developer can accidentally compare against the wrong number. Enums are type-safe: the Java compiler will refuse to compile code that compares an ErrorType against an integer or against a string. They also serve as self-documenting code. When you read ErrorType.InvalidMemoryAddress, you understand the meaning immediately, whereas ERROR_3 forces you to look up a table.
Alternative considered: exception subclasses. One could create a class hierarchy: InvalidRegisterException extends SimulatorException, StackOverflowException extends SimulatorException, and so forth. This is the approach that large frameworks like Spring use. We rejected it because ten separate exception classes for a project of this size would be excessive. The enum approach keeps all error categories in a single file while still providing type-safe discrimination. If the project grows to hundreds of error types, the subclass approach would become more appropriate.

The Three Constructors

    private final ErrorType errorType;
    private final int address;

    public SimulatorException(String message) {
        super(message);
        this.errorType = ErrorType.InvalidData;
        this.address = -1;
    }

    public SimulatorException(String message, ErrorType errorType) {
        super(message);
        this.errorType = errorType;
        this.address = -1;
    }

    public SimulatorException(String message, ErrorType errorType, int address) {
        super(message);
        this.errorType = errorType;
        this.address = address;
    }
What is happening here. Three constructors form a telescoping pattern. The simplest takes only a message and defaults to InvalidData with no address. The middle constructor adds the error type. The fullest constructor adds the memory address where the error occurred. This address field is critical: when the cpu encounters an invalid opcode at address 0x2007, the exception carries that address so the user interface can display exactly where in their programme the failure happened. The sentinel value -1 means "address not applicable."

The toString Override

    @Override
    public String toString() {
        return String.format("[%s]%s: %s",
            errorType,
            (address != -1) ? String.format(" at 0x%04X", address) : "",
            getMessage());
    }
What is happening here. The toString method is what Java calls when the object must be rendered as text (for example, when printing it to the console). We override the default Exception.toString() to produce output like [InvalidOpcode] at 0x2007: Unknown opcode: 0x08. The %04X format specifier prints the address as a four-digit uppercase hexadecimal number with leading zeros, which is the standard notation for 8085 memory addresses. The conditional expression ensures that when no address is relevant (such as for a syntax error during assembly), no spurious "at 0xFFFF" is appended.

3. Memory: The Interface Contract

Memory.java Memory.java 16 lines

In the physical 8085, memory is not part of the CPU. Memory is a separate integrated circuit (or set of circuits) that sits on the circuit board and communicates with the CPU over the address bus (16 wires carrying the address) and the data bus (8 wires carrying the data byte). The CPU sends an address on the address bus, asserts a read or write control signal, and the memory chip responds by either providing or accepting a byte on the data bus.

In our simulator, this separation is modelled by making memory an interface rather than a class. The Architecture class (the CPU) implements this interface, which means the CPU "contains" physical memory in our model. This is a simplification. In a more elaborate simulator, one might have separate RAM and ROM classes, or even memory-mapped I/O regions, all implementing this same interface through the Strategy pattern.

public interface Memory {
    int DEFAULT_START_ADDRESS = 0x0000;
    int DEFAULT_END_ADDRESS   = 0xFFFF;
    int MAX_MEMORY_SIZE       = 65536;

    int  readMemory(int address) throws SimulatorException;
    void writeMemory(int address, int data) throws SimulatorException;
    void resetMemory();
    int[] getMemoryDump();
    int  getMemoryStart();
    int  getMemoryEnd();
    void loadProgram(int startAddress, int[] data) throws SimulatorException;
    void displayMemoryRange(int fromAddress, int toAddress);
}
Why int for addresses and data, not byte? Java's byte type is signed, ranging from -128 to +127. The 8085's memory locations hold values from 0x00 to 0xFF (0 to 255). If we used byte, a value like 0xFF would be interpreted as -1 in Java, requiring constant & 0xFF masking at every usage site. Using int avoids this problem entirely: an int can hold any value from 0 to 255 without sign ambiguity. We still mask with & 0xFF at write boundaries for safety, but we do it once (in the implementation) rather than everywhere.
Why int for addresses, not short? Java's short is 16 bits, which matches the 8085's 16-bit address bus. However, short is signed in Java (-32768 to +32767), whereas memory addresses are inherently unsigned (0x0000 to 0xFFFF). Address 0xFFFF would be interpreted as -1 as a short. Java was not designed with unsigned types (until Java 8 added limited support). Using int for addresses gives us a clean unsigned range from 0 to 65535 with no sign complications. The minor memory overhead (4 bytes instead of 2 per address variable) is negligible in a simulator context.

The Constants

Interface constants in Java. When you declare a variable inside a Java interface, it is implicitly public static final. This means DEFAULT_START_ADDRESS, DEFAULT_END_ADDRESS, and MAX_MEMORY_SIZE are compile-time constants accessible from anywhere as Memory.MAX_MEMORY_SIZE. We place them in the interface rather than in the implementing class because they represent the contract of what any memory implementation must respect, not implementation details of one particular memory.
Alternative: abstract class instead of interface. We could have used an abstract class BaseMemory with default method implementations. This would allow shared code (like the displayMemoryRange method) to live in the base class. We chose an interface because Java supports multiple interface implementation but only single class inheritance. Since Architecture also implements InstrucSet, using interfaces for both keeps the design flexible. If shared method implementations become necessary in the future, Java's default methods in interfaces (available since Java 8) would let us add them without changing the design.

4. ALU: Arithmetic Logic Unit

ALU.java ALU.java 150 lines

The Arithmetic Logic Unit is the computational engine of the 8085. On the physical chip, the ALU is a combinational logic circuit that accepts two 8-bit inputs and produces one 8-bit output along with five condition flags. It does not store the inputs or the output; it merely computes. In our simulator, the ALU class encapsulates both the computation methods and the five flag flip-flops, because in the real hardware those flags are physically located adjacent to the ALU and are set by its output signals.

The Five Flags

public class ALU {
    private boolean signFlag;      // S: set if bit 7 of result is 1
    private boolean zeroFlag;      // Z: set if result is 0x00
    private boolean auxCarryFlag;  // AC: carry from bit 3 to bit 4
    private boolean parityFlag;    // P: set if number of 1-bits is even
    private boolean carryFlag;     // CY: carry/borrow out of bit 7
}
What each flag represents in hardware. The 8085's flag register is a physical 8-bit latch where only five bits are meaningfully used. The layout is [S Z 0 AC 0 P 1 CY], reading from bit 7 down to bit 0. Bit 5 is always 0, bit 3 is always 0, and bit 1 is always 1. These fixed bits are artefacts of the silicon layout and exist because Intel did not wire those flip-flop outputs to anything. We model each meaningful bit as a separate boolean field because booleans read more naturally in conditionals (if (zeroFlag) versus if ((flags & 0x40) != 0)).

The Core Flag Update Method

    public void updateAllFlags(int result, int op1Lower, int op2Lower, boolean isSubtraction) {
        if (isSubtraction) {
            carryFlag = (result < 0);
            auxCarryFlag = (op1Lower - op2Lower) < 0;
        } else {
            carryFlag = (result > 0xFF);
            auxCarryFlag = (op1Lower + op2Lower) > 0x0F;
        }
        int masked = result & 0xFF;
        zeroFlag = (masked == 0);
        signFlag = ((masked & 0x80) != 0);
        parityFlag = (Integer.bitCount(masked) % 2 == 0);
    }
Line-by-line breakdown.
  • result is the full integer result of the operation before masking to 8 bits. For addition, if A=0xFF and we add 0x01, result is 0x100 (256). This exceeds 0xFF, so carryFlag is set. For subtraction, if A=0x05 and we subtract 0x06, result is -1. A negative result means a borrow occurred, so carryFlag is set.
  • op1Lower and op2Lower are the lower nibbles (bits 0 through 3) of the two operands. The auxiliary carry flag is set when there is a carry out of bit 3. For addition, this occurs when the sum of the lower nibbles exceeds 0x0F (15). For subtraction, it occurs when the lower nibble of the first operand is less than that of the second (a borrow is needed).
  • masked = result & 0xFF truncates the result to 8 bits, discarding any carry or borrow information. This is the value that will actually be stored back into the register.
  • The zero flag is true when the masked result is zero. The sign flag copies bit 7. The parity flag uses Integer.bitCount(), a built-in Java method that counts the number of 1-bits in an integer, which is far more readable than a manual loop.
Why does isSubtraction change the carry logic? In digital hardware, subtraction is performed by adding the two's complement of the subtrahend. The carry flag has opposite semantics for addition versus subtraction: for addition, a carry out of bit 7 sets CY; for subtraction, a borrow sets CY (which corresponds to the result going negative). Some architectures (like ARM) invert CY after subtraction; the 8085 sets CY=1 when a borrow occurs, meaning the result is negative. Our code uses the sign of the Java int result (negative means borrow) as a direct indicator.

The Preserve-Carry Variant

    public void updateFlagsExceptCarry(int result, int op1Lower, int op2Lower, boolean isSubtraction) {
        boolean savedCy = carryFlag;
        updateAllFlags(result, op1Lower, op2Lower, isSubtraction);
        carryFlag = savedCy;
    }
What is happening here. The 8085 specification states that the INR (increment) and DCR (decrement) instructions update all flags except the carry flag. This is a hardware design choice: Intel wanted increment and decrement to be usable as loop counters without disturbing the carry flag that might be needed for multi-byte arithmetic elsewhere. Rather than duplicating the entire flag computation logic with one line removed, we save the carry flag, call the full update, and then restore it. This is a classic example of the Template Method principle in miniature.

The Add and Subtract Methods

    public int add(int a, int b, int carryIn) {
        int result = a + b + carryIn;
        updateAllFlags(result, a & 0x0F, (b + carryIn) & 0x0F, false);
        return result & 0xFF;
    }

    public int subtract(int a, int b, int borrowIn) {
        int result = a - b - borrowIn;
        updateAllFlags(result, a & 0x0F, (b + borrowIn) & 0x0F, true);
        return result & 0xFF;
    }
The carryIn parameter. This single parameter is what distinguishes ADD from ADC (Add with Carry) and SUB from SBB (Subtract with Borrow). When the CPU executes ADD, it calls alu.add(a, operand, 0). When it executes ADC, it calls alu.add(a, operand, alu.isCarryFlag() ? 1 : 0). The carry from a previous operation is fed back in as the third operand. This is exactly how multi-byte arithmetic works: you ADD the low bytes, then ADC the high bytes so that any carry from the low addition propagates upward.

Logical Operations

    public int and(int a, int b) {
        int result = (a & b) & 0xFF;
        carryFlag = false;
        auxCarryFlag = true;
        updateSzp(result);
        return result;
    }
Why does AND set AC=1 and CY=0? This is a documented quirk of the 8085. The AND instruction always clears the carry flag (because AND cannot produce a carry in any mathematical sense) and always sets the auxiliary carry flag. The reason for setting AC is historical: Intel's implementation of ANA used an internal circuit path that happens to generate the AC signal. Some later Intel documentation explains it as "the 8085 effectively ORs the bit-3 outputs of the two operands during ANA," but in practice, programmers simply memorise the rule. For OR and XOR, both CY and AC are cleared, because those operations use a different internal circuit path.

The Flag Register Byte

    public int getFlagsByte() {
        int flags = 0x02;          // bit 1 always 1
        if (signFlag)     flags |= 0x80;
        if (zeroFlag)     flags |= 0x40;
        if (auxCarryFlag) flags |= 0x10;
        if (parityFlag)   flags |= 0x04;
        if (carryFlag)    flags |= 0x01;
        return flags;
    }
What is happening here. This method composes the five boolean flags into the 8-bit flag register format used by the PUSH PSW and POP PSW instructions. The initial value 0x02 (binary 00000010) sets bit 1 to 1, matching the real 8085's behaviour where this bit is always high. Each |= operation sets the corresponding bit if the flag is true. The inverse method, setFlagsByte, decomposes an 8-bit value back into five booleans using bitwise AND with masks.

5. InstrucSet: The Instruction Set Interface

InstrucSet.java InstrucSet.java 94 lines

The InstrucSet interface is the formal specification of every instruction the 8085 can execute. It contains 74 method signatures, one for each unique mnemonic in the 8085 instruction set. The 8085 has approximately 246 distinct opcodes (different register combinations produce different opcodes), but many of those opcodes share the same mnemonic. For example, MOV A,B (opcode 0x78) and MOV A,C (opcode 0x79) are different opcodes but the same mnemonic, MOV, with different parameters. Our design decision was to represent each mnemonic as one method with string parameters for registers. This produces 74 methods instead of 246, which is dramatically cleaner.

Data Transfer Group

    void mov(String dest, String src) throws SimulatorException;   // MOV dest,src: copy src to dest
    void mvi(String dest, int data) throws SimulatorException;     // MVI dest,d8: load immediate byte
    void lxi(String regPair, int data16) throws SimulatorException;// LXI rp,d16: load 16-bit immediate
    void lda(int address) throws SimulatorException;               // LDA addr: A = mem[addr]
    void sta(int address) throws SimulatorException;               // STA addr: mem[addr] = A
The parameter design philosophy. Each method signature mirrors the assembly syntax. mov("A", "B") corresponds to the assembly instruction MOV A, B. When the operand is a register, we use String. When it is an immediate data byte, we use int. When it is a 16-bit address, we also use int (since Java's int can hold any 16-bit value). The special string "M" represents the memory location pointed to by the HL register pair, which the real 8085 encodes as register code 6 in the opcode byte.

Arithmetic Group

    void add(String src) throws SimulatorException;   // ADD src: A = A + src, all flags
    void adc(String src) throws SimulatorException;   // ADC src: A = A + src + CY
    void inr(String dest) throws SimulatorException;  // INR dest: dest++, CY unchanged
    void inx(String regPair) throws SimulatorException;// INX rp: rp++, no flags
    void dad(String regPair) throws SimulatorException;// DAD rp: HL = HL + rp, only CY
The flag specification in comments. Each comment succinctly states which flags are affected. This is critical documentation. INR says "CY unchanged" because it updates S, Z, AC, P but not CY. INX says "no flags" because it performs a 16-bit increment that affects no flags at all. DAD says "only CY" because the 16-bit addition only affects the carry flag. These distinctions matter enormously in real 8085 programming. A common beginner mistake is assuming that INR sets the carry flag, which leads to incorrect conditional jump behaviour.

Branching Group

    void jmp(int address) throws SimulatorException;  // JMP addr: unconditional
    void jc(int address) throws SimulatorException;   // JC: if CY=1
    void jnc(int address) throws SimulatorException;  // JNC: if CY=0
    void call(int address) throws SimulatorException; // CALL: push PC+3, jump
    void ret() throws SimulatorException;             // RET: pop PC from stack
    void rst(int n) throws SimulatorException;        // RST n: call n*8
The conditional call and return pattern. The 8085 has 8 conditional jump variants, 8 conditional call variants, and 8 conditional return variants, all testing the same four flags in positive and negative form (CY/no CY, Z/no Z, S positive/S negative, P even/P odd). Rather than defining 24 methods with complex logic, each conditional variant is a one-line method that checks a flag and delegates to the unconditional version. For example, jc(address) is implemented as if (alu.isCarryFlag()) programCounter = address;. This delegation pattern keeps the interface clean and the implementation DRY.

6. Day 1 Summary and Forward Context

Today we examined the four foundational elements that every other component depends upon:

  1. SimulatorException provides typed, addressed error reporting that mirrors hardware fault signals.
  2. Memory defines the contract for how bytes enter and leave the system, keeping the CPU and memory logically separable.
  3. ALU encapsulates all arithmetic, logical, and comparison operations with correct five-flag semantics, including the delicate edge cases of INR/DCR (carry preserved) and ANA (AC forced high).
  4. InstrucSet formalises the 74-mnemonic instruction set as a Java interface, establishing the contract that the Architecture class must fulfil.

Tomorrow, in Day 2, we will examine how these contracts are consumed: the OpcodeTable that maps all 256 bytes to mnemonics, the Assembler that translates human-readable assembly into those bytes, and the Architecture class that implements every instruction method and drives the fetch-decode-execute cycle.

7. OpcodeTable: The Instruction Decoder ROM

OpcodeTable.java OpcodeTable.java 200 lines

In the physical 8085, when the CPU fetches a byte from memory, that byte enters the instruction register and is decoded by a complex network of logic gates that determines which internal control signals to activate. This decoding network is essentially a hardwired lookup: for any given 8-bit input (256 possible values), there is exactly one set of control signals that will be generated. Our OpcodeTable is the software equivalent of that decoder network. It is a static lookup table that maps every possible opcode byte (0x00 through 0xFF) to its human-readable mnemonic, its byte size (how many bytes the instruction occupies in memory), and its functional category.

The OpcodeEntry Inner Class

public static class OpcodeEntry {
    public final int opcode;
    public final String mnemonic;
    public final int byteSize;
    public final String category;

    public OpcodeEntry(int opcode, String mnemonic, int byteSize, String category) {
        this.opcode = opcode;
        this.mnemonic = mnemonic;
        this.byteSize = byteSize;
        this.category = category;
    }
}
What is happening here. The OpcodeEntry is a value object: an immutable bundle of four pieces of information about a single opcode. All fields are public final, meaning they can be read directly but never modified after construction. This is an intentional trade-off: for a data class with only immutable fields, getters add verbosity without adding safety. In a mutable object, encapsulation through getters/setters would be important. Here, finality provides the same guarantee with less code.
Why String for category, not an enum? Categories like "DATA_TRANSFER", "ARITHMETIC", "LOGICAL", "BRANCHING", and "STACK_IO" are used primarily for display and grouping, not for control flow decisions. An enum would be more type-safe, but the categories are never compared programmatically in the current codebase. If future features like "show only arithmetic instructions" are added, converting to an enum would be a straightforward refactor. For now, strings keep the opcode registration calls more readable.

Loop-Based Table Generation

The 8085 has 246 valid opcodes. Writing 246 individual addEntry calls would be tedious and error-prone. The key insight is that Intel designed the opcode encoding with regular patterns. Certain bit fields within the opcode byte correspond directly to register codes:

MOV Generation: The Nested Loop

// Data Transfer — MOV (63 variants, skip 0x76 = HLT)
for (int d = 0; d < 8; d++)
    for (int s = 0; s < 8; s++) {
        int opcode = 0x40 | (d << 3) | s;
        if (opcode == 0x76) continue;
        addEntry(opcode, "MOV " + REG_NAMES[d] + "," + REG_NAMES[s], 1, "DATA_TRANSFER");
    }
The opcode encoding formula. Every MOV instruction has the binary format 01DDDSSS, where DDD is the 3-bit destination register code and SSS is the 3-bit source register code. The base value 0x40 is 01000000 in binary. The expression (d << 3) shifts the destination code into bits 5-3, and | s places the source code into bits 2-0. This produces all 64 combinations (8 × 8), but the combination where both destination and source are M (code 6) gives opcode 0x76, which Intel assigned to HLT instead. The continue statement skips this one collision.
Why does MOV M,M not exist? The instruction MOV M,M would mean "copy the byte at address HL to the same address HL"—a no-operation that is also pointless. Intel repurposed this opcode slot for HLT, which is far more useful. This is a common trick in ISA design: waste slots in the opcode space are repurposed for high-value instructions. The 8086 and later processors continued this tradition.

Arithmetic Group Generation

// Arithmetic — ADD/ADC/SUB/SBB (8 variants each)
for (int s = 0; s < 8; s++) {
    addEntry(0x80 | s, "ADD " + REG_NAMES[s], 1, "ARITHMETIC");
    addEntry(0x88 | s, "ADC " + REG_NAMES[s], 1, "ARITHMETIC");
    addEntry(0x90 | s, "SUB " + REG_NAMES[s], 1, "ARITHMETIC");
    addEntry(0x98 | s, "SBB " + REG_NAMES[s], 1, "ARITHMETIC");
}
The pattern. ADD opcodes start at 0x80 (10000SSS), ADC at 0x88 (10001SSS), SUB at 0x90 (10010SSS), SBB at 0x98 (10011SSS). The lower three bits encode the source register. One loop body generates 32 entries (4 instruction types × 8 registers). Without the loop, we would need 32 separate addEntry calls. The same pattern extends to ANA/XRA/ORA/CMP (starting at 0xA0/0xA8/0xB0/0xB8), which produces another 32 entries in a single loop.

The Dual HashMap Design

private static final HashMap<Integer, OpcodeEntry> table = new HashMap<>();
private static final HashMap<String, Integer> mnemonicMap = new HashMap<>();
Two-way lookup. The table maps opcode → entry (used by the CPU during execution: "what does byte 0x3E mean?"). The mnemonicMap maps mnemonic → opcode (used by the assembler: "what byte does MVI A produce?"). Both are populated by the same addEntry method, ensuring they are always consistent. This is the Bidirectional Map pattern, implemented manually because Java's standard library does not include one (Guava does, via BiMap).
Alternative: array instead of HashMap. Since opcodes are integers from 0 to 255, a simple OpcodeEntry[256] array would provide O(1) lookup with lower overhead than a HashMap (no hashing, no collision handling). We chose HashMap because it makes the mnemonic-to-opcode reverse lookup natural. An array approach would require a separate data structure or linear scan for mnemonic lookup. For 256 entries, the performance difference is negligible.

8. Assembler: Two-Pass Translation

Assembler.java Assembler.java 270 lines

An assembler is the programme that translates human-readable assembly language (like MVI A, 42H) into machine code bytes (like 3E 42) that the CPU can execute. Our assembler uses the classical two-pass algorithm, which is the same approach used by real assemblers like NASM, MASM, and the original Intel ASM85.

Why Two Passes Are Necessary

Consider this programme:

        JMP DONE     ; Jump to the label DONE
        MVI A, 05H
DONE:   HLT
The forward reference problem. When the assembler encounters JMP DONE on line 1, it needs to know the address of DONE. But DONE is defined on line 3. The assembler has not reached line 3 yet. A single-pass assembler cannot resolve this forward reference. The solution is two passes: Pass 1 scans the entire programme to discover all labels and their addresses, building a "symbol table." Pass 2 generates machine code, and when it encounters JMP DONE, it looks up DONE in the symbol table to find the target address.

Pass 1: Label Collection

int currentAddress = startAddress;
for (String line : lines) {
    line = stripComment(line).trim();
    if (line.isEmpty()) continue;
    if (line.contains(":")) {
        int idx = line.indexOf(':');
        symbolTable.put(line.substring(0, idx).trim().toUpperCase(), currentAddress);
        line = line.substring(idx + 1).trim();
        if (line.isEmpty()) continue;
    }
    currentAddress += getInstructionSize(line);
}
Line-by-line.
  • stripComment(line) removes everything after a semicolon. In 8085 assembly, ; begins a comment, just like // in Java or # in Python.
  • If a line contains a colon, everything before the colon is a label name. The label is stored in the symbol table mapped to the current address. If the line also contains an instruction after the colon (like LOOP: DCR B), the instruction is processed normally.
  • getInstructionSize(line) determines how many bytes the instruction occupies (1, 2, or 3) based on the mnemonic. The current address is incremented by this amount. No machine code is generated in this pass.

The Instruction Size Logic

private int getInstructionSize(String line) {
    String mnemonic = line.trim().split("\\s+")[0].toUpperCase();
    switch (mnemonic) {
        case "LXI": case "LDA": case "STA": case "LHLD": case "SHLD":
        case "JMP": case "JC":  case "JNC": case "JZ":  case "JNZ":
        case "CALL": /* ... all 3-byte mnemonics ... */
            return 3;
        case "MVI": case "ADI": case "ACI": case "SUI": case "SBI":
        case "ANI": case "ORI": case "XRI": case "CPI":
        case "IN":  case "OUT":
            return 2;
        default:
            return 1;
    }
}
The 8085 instruction length rule. Every 8085 instruction occupies exactly 1, 2, or 3 bytes. Single-byte instructions (like MOV, ADD, NOP, HLT) encode everything in the opcode itself. Two-byte instructions (like MVI A, 42H) have an opcode byte followed by one data byte. Three-byte instructions (like JMP 2050H, LXI H, 3000H) have an opcode byte followed by two address/data bytes in little-endian order (low byte first, high byte second). No 8085 instruction exceeds 3 bytes.

Pass 2: Machine Code Generation

case "MOV": return new int[]{0x40 | (regCode(ops[0]) << 3) | regCode(ops[1])};

case "MVI": return new int[]{0x06 | (regCode(ops[0]) << 3), parseImm8(ops[1])};

case "LXI": return i3(0x01 | (rpCode(ops[0]) << 4), parseImm16(ops[1]));
Opcode computation vs. table lookup. Rather than looking up the opcode in the OpcodeTable, the assembler computes opcodes using the same bit-field encoding rules that Intel's hardware uses. For MOV A, B: regCode("A") returns 7, regCode("B") returns 0, so the opcode is 0x40 | (7 << 3) | 0 = 0x40 | 0x38 | 0x00 = 0x78. This approach is more maintainable than a 63-entry lookup table for MOV alone, and it mirrors how the hardware decoder works in reverse.

The Number Parser

private int parseNumber(String text) throws SimulatorException {
    text = text.trim().toUpperCase();
    try {
        if (text.endsWith("H"))     return Integer.parseInt(text.substring(0, text.length() - 1), 16);
        if (text.startsWith("0X"))  return Integer.parseInt(text.substring(2), 16);
        if (text.endsWith("B"))     return Integer.parseInt(text.substring(0, text.length() - 1), 2);
        return Integer.parseInt(text);
    } catch (NumberFormatException e) {
        throw new SimulatorException("Invalid number format: " + text,
                SimulatorException.ErrorType.SyntaxError);
    }
}
Multi-format number parsing. The 8085 assembly convention supports several numeric formats. 42H is hexadecimal (the trailing H). 0x2A is also hexadecimal (the C-language prefix). 01000010B is binary (the trailing B). Plain digits like 66 are decimal. The parser tries each format in order. The suffix-based hex format (42H) is checked first because it is the most common format in 8085 assembly. The 0x prefix is a modern convenience. Binary is useful for students learning bit manipulation.
Why the priority order matters. If we checked for decimal first, the string 42H would fail the decimal parse and reach the hex check as a fallback, which would still work. But consider 10B: if we checked decimal first, it would parse as 10 (decimal), not 2 (binary). By checking the B suffix before decimal, we ensure binary notation takes priority for strings that end with B. Hexadecimal also takes priority over binary (0ABH is hex, not binary with an odd suffix). This priority chain mirrors the parsing rules of real assemblers.

The Address Resolution Strategy

private int resolveAddress(String ref) throws SimulatorException {
    try {
        return parseImm16(ref);
    } catch (SimulatorException e) {
        String label = ref.trim().toUpperCase();
        if (symbolTable.containsKey(label)) return symbolTable.get(label);
        throw new SimulatorException("Undefined label: " + ref,
                SimulatorException.ErrorType.UndefinedAddress);
    }
}
Numeric-first resolution. When the assembler encounters JMP 2050H, the operand 2050H is a numeric address. When it encounters JMP LOOP, the operand LOOP is a symbolic label. The resolver tries to parse the operand as a number first. If that fails (because "LOOP" is not a valid number), it looks the operand up in the symbol table. If both fail, the operand is an undefined label—a meaningful error. This approach is simpler and faster than trying to distinguish labels from numbers syntactically (which is complex because labels can contain digits).

9. Architecture: The CPU Core

Architecture.java Architecture.java 853 lines

The Architecture class is the heart of the simulator. It implements both the Memory interface and the InstrucSet interface, meaning it is simultaneously the memory subsystem and the instruction executor. In hardware terms, it is the entire CPU die plus the memory chips, combined into one object.

State Variables

private HashMap<String, Integer> generalRegisters; // B, C, D, E, H, L
private HashMap<String, Integer> specialRegisters; // A, W, Z

Why HashMap for registers, not an array? The readability argument. With a HashMap, instruction implementations read getRegister("A") and setRegister("B", value). With an array indexed by constants, they would read registers[REG_A] and registers[REG_B] = value. Both work, but the HashMap version is self-documenting at every call site. The performance penalty (HashMap vs. array indexing) is negligible for a simulator that processes at most tens of thousands of instructions per second, not millions. In a performance-critical emulator (like a game console emulator running at 60fps), array indexing would be essential.

The Fetch-Decode-Execute Cycle

The most critical piece of code in the entire project:

private int fetchByte() throws SimulatorException {
    int data = readMemory(programCounter);
    programCounter = (programCounter + 1) & 0xFFFF;
    return data;
}

private int fetchWord() throws SimulatorException {
    int low = fetchByte();
    int high = fetchByte();
    return (high << 8) | low;
}
The fetch mechanism. fetchByte() reads one byte from the memory location pointed to by the program counter, then increments PC. The & 0xFFFF mask ensures PC wraps around from 0xFFFF to 0x0000, mimicking how a 16-bit counter overflows in hardware. fetchWord() fetches two bytes: the low byte first, then the high byte. This is little-endian byte order—the 8085 stores the least significant byte at the lower address. The word is reconstructed by shifting the high byte left by 8 and ORing with the low byte.

The Instruction Dispatch Switch

private void executeInstruction() throws SimulatorException {
    int opcode = fetchByte();
    switch (opcode) {
        case 0x00: nop(); break;
        case 0x01: lxi("B", fetchWord()); break;
        // ... 200+ more cases ...
        case 0x76: hlt(); break;
        // ...
        default:
            if (opcode >= 0x40 && opcode <= 0x7F) {
                String[] regNames = {"B","C","D","E","H","L","M","A"};
                int dest = (opcode >> 3) & 0x07;
                int src  = opcode & 0x07;
                mov(regNames[dest], regNames[src]);
            } else {
                throw new SimulatorException("Unknown opcode", ...);
            }
    }
}
Why a giant switch, not a lookup table of lambdas? In a modern Java codebase, you might create a Map<Integer, Runnable> and populate it with lambda expressions for each opcode. That approach is elegant but introduces overhead from autoboxing, lambda capture, and indirection. The switch statement compiles to a tableswitch bytecode in Java, which is essentially a jump table—the fastest possible dispatch mechanism on the JVM. For a simulator where this method is called for every single instruction, raw speed matters.

The MOV instructions (63 opcodes occupying the range 0x40-0x7F minus 0x76) are handled in the default branch using bit extraction rather than 63 individual cases. This is the same encoding trick the hardware uses: extract bits 5-3 for the destination and bits 2-0 for the source.

An Instruction Implementation: CALL and RET

@Override
public void call(int address) throws SimulatorException {
    int returnAddr = programCounter;
    stackPointer = (stackPointer - 1) & 0xFFFF;
    writeMemory(stackPointer, (returnAddr >> 8) & 0xFF);
    stackPointer = (stackPointer - 1) & 0xFFFF;
    writeMemory(stackPointer, returnAddr & 0xFF);
    programCounter = address & 0xFFFF;
}

@Override
public void ret() throws SimulatorException {
    int low = readMemory(stackPointer);
    stackPointer = (stackPointer + 1) & 0xFFFF;
    int high = readMemory(stackPointer);
    stackPointer = (stackPointer + 1) & 0xFFFF;
    programCounter = (high << 8) | low;
}
How the stack works. The 8085's stack grows downward in memory. SP starts at a high address (we default to 0xFFFF) and decreases with each PUSH or CALL. During CALL:
  1. By the time CALL executes, PC has already advanced past the 3-byte CALL instruction, so the current PC is the return address.
  2. SP is decremented, and the high byte of the return address is written to the new SP location.
  3. SP is decremented again, and the low byte is written. (Little-endian: low byte at lower address.)
  4. PC is set to the target address.
During RET, the inverse happens: the low byte is read from [SP], SP increments; the high byte is read from [SP], SP increments; PC is reconstructed.

The Conditional Branch Pattern

@Override public void jc(int address)  { if (alu.isCarryFlag()) programCounter = address; }
@Override public void jnc(int address) { if (!alu.isCarryFlag()) programCounter = address; }
@Override public void cc(int address) throws SimulatorException  { if (alu.isCarryFlag()) call(address); }
@Override public void cnc(int address) throws SimulatorException { if (!alu.isCarryFlag()) call(address); }
@Override public void rc() throws SimulatorException   { if (alu.isCarryFlag()) ret(); }
@Override public void rnc() throws SimulatorException  { if (!alu.isCarryFlag()) ret(); }
Delegation over duplication. Each conditional variant is a guard clause that delegates to the unconditional implementation. cc (Call if Carry) checks the carry flag; if set, it calls call(address); if not, it does nothing (the programme continues at the next instruction). This produces 24 near-identical one-line methods (8 for jump, 8 for call, 8 for return), which may seem verbose but is maximally readable: every method does exactly one thing, and the flag condition is visible at a glance.

10. Day 2 Summary

Today we examined the three files that form the execution pipeline:

  1. OpcodeTable maps all 256 opcodes using loop-generated entries that exploit the encoding regularity of the 8085 ISA. It provides O(1) bidirectional lookup between opcodes and mnemonics.
  2. Assembler implements a classical two-pass algorithm: Pass 1 builds the symbol table by scanning labels and computing addresses; Pass 2 generates machine code by computing opcodes from bit fields and resolving labels from the symbol table. It supports hex, binary, and decimal number formats.
  3. Architecture is the CPU itself. It holds registers in HashMaps, memory in an int array, and delegates computation to the ALU. The fetch-decode-execute cycle is a switch statement that dispatches all 256 opcodes, with the 63 MOV variants handled by bit extraction in the default branch.

The Step-Through Debugger

private void handleStepThrough() {
    while (!architecture.isHalted()) {
        int pc = architecture.getProgramCounter();
        int opcode = architecture.readMemory(pc);
        System.out.printf("[%04XH] %s\n", pc, architecture.disassemble(opcode));
        architecture.step();
        System.out.println(architecture.getCPUState());

        String input = scanner.nextLine().trim();
        if (input.equalsIgnoreCase("q")) break;
    }
}
What stepping looks like. Before each instruction, the debugger shows the current PC address and the disassembled instruction at that address. Then it executes one instruction via architecture.step() and displays the complete CPU state (all registers, flags). The user presses Enter to advance to the next instruction or types 'q' to return to the menu. This mirrors the single-step mode on a real 8085 trainer kit, where you press a "step" button and observe the register state on the LED display after each instruction.

14. Working Example Programmes

Example 1: Add Two Numbers

MVI A, 05H    ; Load 5 into accumulator
MVI B, 03H    ; Load 3 into B
ADD B         ; A = A + B = 8
STA 3000H     ; Store result at memory address 3000H
HLT           ; Stop
Machine code trace. This programme assembles to 8 bytes: 3E 05 06 03 80 32 00 30 76. The assembler processes MVI A as opcode 0x3E (from the formula 0x06 | (7 << 3) where 7 is the register code for A) followed by the immediate byte 0x05. ADD B is opcode 0x80 (register B = code 0). STA 3000H is opcode 0x32 followed by 0x00 (low byte of 3000H) and 0x30 (high byte)—little-endian order.

Example 2: Multiply by Repeated Addition

        MVI B, 04H    ; Multiplicand
        MVI C, 03H    ; Multiplier (loop count)
        MVI A, 00H    ; Clear accumulator
LOOP:   ADD B         ; A = A + B
        DCR C         ; C = C - 1
        JNZ LOOP      ; If C != 0, repeat
        STA 3000H     ; Store product
        HLT
Label resolution in action. During Pass 1, the assembler assigns addresses: MVI B is at 2000H (2 bytes), MVI C at 2002H (2 bytes), MVI A at 2004H (2 bytes), LOOP is at 2006H. The label LOOP maps to address 0x2006 in the symbol table. During Pass 2, when JNZ LOOP is encountered, the assembler resolves LOOP to 0x2006 and produces the 3-byte instruction C2 06 20 (opcode 0xC2 for JNZ, followed by address 0x2006 in little-endian). The loop executes 3 times, accumulating 4+4+4=12 (0x0C) in the accumulator.

Example 3: Find Larger of Two Numbers

        MVI A, 08H
        MVI B, 05H
        CMP B         ; Compare A with B (sets flags from A-B)
        JNC DONE      ; If A >= B (no carry), A is already larger
        MOV A, B      ; Otherwise, B is larger, copy to A
DONE:   STA 3000H
        HLT
How CMP drives conditional logic. CMP B internally computes A - B and sets flags, but does NOT store the result in A. If A=8 and B=5, then A-B=3 (positive, no borrow), so CY=0. JNC ("jump if no carry") sees CY=0 and jumps to DONE, skipping the MOV. If A were smaller than B, the subtraction would produce a borrow (CY=1), JNC would not jump, and MOV A,B would overwrite A with the larger value. This is the fundamental comparison pattern for the 8085.

15. Full-Fledged Programme Roadmap

The current simulator is a working educational tool, but a production-grade 8085 simulator would require dozens of additional features. Below is a comprehensive roadmap organised into six development phases, with detailed descriptions of what each feature entails, why it matters, and how it could be implemented.

Phase 1 File I/O and Programme Management

Enable loading and saving assembly files, eliminating the need to retype programmes.

1.1 Assembly File Import (.asm) High Priority

The ability to load assembly source code from .asm files on disk. The user would specify a file path, and the simulator would read the file contents, pass them to the assembler, and load the resulting machine code into memory. This eliminates the tedious process of typing programmes line by line into the console. Implementation involves adding a java.io.File reader in UserInterface and a new menu option. File encoding should default to UTF-8 with fallback to ASCII for compatibility with legacy assembly files.

1.2 Assembly File Export (.asm) High Priority

Save the current programme in memory back to an .asm file using a disassembler. This requires implementing a disassembleRange(int from, int to) method in Architecture that walks through memory, decodes each opcode using OpcodeTable, and reconstructs the mnemonic with operands. The disassembler must correctly determine instruction boundaries (consuming 1, 2, or 3 bytes per instruction) and format addresses and data in hex notation.

1.3 Machine Code Binary Export (.bin / .hex) Medium Priority

Export the assembled machine code as a raw binary file or in Intel HEX format (.hex). Intel HEX is an ASCII text representation of binary data with checksums, widely used for programming EPROM chips. Each line starts with a colon and contains a byte count, address, record type, data bytes, and a checksum. Supporting Intel HEX output would allow students to programme real 8085 trainer kits with code developed in the simulator.

1.4 Session Save/Restore Low Priority

Serialise the entire simulator state (registers, memory, flags, PC, SP, I/O ports, symbol table) to a JSON or binary file, and restore it later. This enables pausing a debugging session and resuming it hours or days later. Java's built-in Serializable interface could handle this, but JSON is more portable and human-readable. Libraries like Gson or Jackson would be used for serialisation.

Phase 2 Graphical User Interface

Replace or supplement the CLI with a visual interface for real-time inspection.

2.1 JavaFX or Swing Desktop GUI High Priority

A graphical interface with panels for the code editor (with syntax highlighting), register view, flag view, memory hex dump, and stack display. JavaFX provides modern styling with CSS-like theming, while Swing offers platform-native look and feel. The architecture already exposes getters for all state (getRegister(), getCPUState(), getMemoryDump()), so the GUI would only consume existing APIs without modifying core logic. The GUI should use an MVC or MVVM pattern to remain decoupled from the simulation engine.

2.2 Real-Time Memory Viewer High Priority

A scrollable hex-dump panel that highlights the bytes that changed during the last instruction execution. This requires tracking a "dirty set" of modified addresses. Before each instruction, the simulator would snapshot the addresses that write operations target. After execution, the GUI would highlight those cells in a different colour. Implementation: add an ArrayList<Integer> dirtyAddresses field to Architecture that writeMemory() appends to, cleared at the start of each instruction cycle.

2.3 Register Animation Low Priority

Animate data flow between registers during instruction execution. For example, when MOV B, A executes, a visual arrow or data pulse travels from the A register box to the B register box. This would be purely cosmetic but immensely helpful for students who are still building mental models of how data moves inside a CPU. Implementation would use JavaFX animation timelines.

2.4 Web-Based Interface Medium Priority

Expose the simulator as a REST API (using Spring Boot or Javalin) and build a web frontend with HTML/CSS/JavaScript. The REST API would have endpoints like POST /assemble, POST /run, POST /step, and GET /state. The web frontend would use WebSockets for real-time state updates during execution. This would make the simulator accessible without installing Java, ideal for classroom settings where students use school computers with restricted software installation rights.

Phase 3 Assembler Enhancement

Extend the assembler to support professional-grade assembly language features.

3.1 Assembler Directives (ORG, DB, DW, DS, EQU) High Priority

ORG 3000H sets the assembly origin address. DB 42H, 55H (Define Byte) places raw data bytes into memory. DW 2050H (Define Word) places a 16-bit value in little-endian order. DS 10 (Define Storage) reserves 10 bytes of uninitialised space. COUNTER EQU 05H assigns a constant value to a symbol. These directives are essential for any non-trivial assembly programme. Implementation requires extending the assembler's parsing to recognise these pseudo-instructions and handle them in both Pass 1 (for address computation) and Pass 2 (for data generation).

3.2 Macro Support Medium Priority

Define reusable code templates: MACRO ADD16 ... ENDM. When the assembler encounters a macro invocation, it expands the template with the provided arguments. This requires a pre-processing pass (before the standard two passes) that scans for MACRO definitions, stores them in a macro table, and performs textual substitution. Macros can be nested but not recursive (to avoid infinite expansion).

3.3 Include Files Medium Priority

INCLUDE "stdlib.asm" would insert the contents of another file at the current position. This enables code reuse: common subroutines (like BCD arithmetic or string handling) can be placed in library files and included in multiple programmes. The assembler would resolve include paths relative to the main file's directory, with protection against circular includes.

3.4 Error Position Reporting (Line Numbers) High Priority

Currently, assembly errors report the error type and message but not the line number in the source code. Adding line-number tracking requires carrying a line counter through both assembly passes and embedding it in thrown SimulatorException objects. The error output would change from [SyntaxError]: Unknown mnemonic: MVIX to [SyntaxError] at line 7: Unknown mnemonic: MVIX.

3.5 Listing File Generation Low Priority

Generate a .lst file that shows each line of source code alongside its assembled address and machine code bytes. For example: 2000 3E 05 MVI A, 05H. Listing files are invaluable for debugging and are standard output of professional assemblers. This would be a side-effect of Pass 2: as each instruction is assembled, the listing line is formatted and appended to an output buffer.

Phase 4 Debugging and Analysis Tools

Professional-grade debugging features for serious programme analysis.

4.1 Breakpoints High Priority

Set breakpoints at specific memory addresses. During runFrom(), the execution loop checks whether the current PC matches any breakpoint address before executing the instruction. If it does, execution pauses and control returns to the user. Implementation: add a HashSet<Integer> breakpoints field to Architecture. The check costs O(1) per instruction due to the HashSet's hash-based lookup, which is negligible overhead.

4.2 Watchpoints (Memory/Register Watches) Medium Priority

Pause execution when a specific memory address or register value changes. This is more expensive than breakpoints because every write operation must be checked against the watchpoint list. Implementation: wrap writeMemory() and setRegister() with watchpoint checks. Use a HashMap<Integer, Integer> to track watched addresses and their previous values.

4.3 Execution Trace / History Log Medium Priority

Record the PC, opcode, and register state at every instruction execution into a circular buffer. The user can then replay the execution history, step backwards, and identify where a bug was introduced. The buffer should be bounded (e.g., last 10,000 instructions) to prevent memory exhaustion on long-running programmes.

4.4 Stack Frame Visualiser Low Priority

Display the stack as a labelled structure showing return addresses, saved register pairs, and subroutine nesting depth. This requires the simulator to track CALL/RET relationships and annotate stack entries with the source CALL address. Useful for understanding how nested subroutines work.

4.5 Instruction Cycle Counter Medium Priority

Count the total number of T-states (clock cycles) consumed by the programme. Each 8085 instruction has a defined T-state count (e.g., MOV r,r = 4 T-states, MVI r,d8 = 7 T-states, CALL = 18 T-states). Add a tStates field to OpcodeEntry and accumulate the count during execution. This enables performance analysis and is essential for timing-sensitive applications.

4.6 Comparison / Diff Mode Low Priority

Run two programmes side by side and highlight differences in register and memory state after execution. Useful for verifying that an optimised version of a programme produces the same results as the original. Implementation would instantiate two Architecture objects and compare their states.

Phase 5 Hardware Accuracy and Advanced Emulation

Bring the simulator closer to the behaviour of real silicon.

5.1 Full Interrupt System (RST 5.5, 6.5, 7.5, TRAP) High Priority

The 8085 has five hardware interrupts: TRAP (non-maskable, highest priority), RST 7.5 (edge-triggered), RST 6.5 (level-triggered), RST 5.5 (level-triggered), and INTR (general). Currently, ei() and di() set a boolean but nothing tests it. A full implementation would include an interrupt pending queue, priority resolution, interrupt masking via SIM/RIM, and vector address computation. The RIM instruction would return the actual interrupt mask, pending interrupts, and SID (serial input data) bit.

5.2 Serial I/O (SID/SOD) Medium Priority

The 8085 has dedicated serial input (SID) and serial output (SOD) pins. SIM writes the SOD bit; RIM reads the SID bit. A simulator implementation could connect SID/SOD to a console character stream, or to a virtual UART that communicates over a network socket. This would allow the simulator to interact with external programmes via serial protocol.

5.3 Memory-Mapped I/O Medium Priority

Map certain memory addresses to virtual hardware devices. For example, addresses 0xFF00-0xFF0F could be mapped to a virtual seven-segment display, where writing a byte to 0xFF00 causes a digit to appear on a simulated display widget. Implementation: add a Map<IntRange, IODevice> to Architecture that readMemory() and writeMemory() check before falling through to the regular memory array.

5.4 Clock Speed Simulation Low Priority

The real 8085 runs at 3-5 MHz, executing one T-state per clock cycle. A clock speed simulation would pause between instructions to match real-time execution speed. At 3.072 MHz, each T-state is approximately 325 nanoseconds. A MOV instruction (4 T-states) would take 1.3 microseconds. Implementation: use Thread.sleep() with nanosecond precision (or LockSupport.parkNanos()) between instructions, scaled by the T-state count of each instruction.

5.5 Undocumented Opcode Support Low Priority

The 8085 has 10 undocumented opcodes (0x08, 0x10, 0x18, 0x28, 0x38, 0xCB, 0xD9, 0xDD, 0xED, 0xFD) that were discovered through reverse-engineering. Some perform useful operations like DSUB (16-bit subtract) and RSTV (restart on overflow). Supporting these would make the simulator useful for studying the 8085's hidden capabilities. References: Ken Shirriff's die analysis blog and Dehnadi's undocumented opcode documentation.

5.6 ROM/RAM Region Configuration Medium Priority

Mark certain memory ranges as read-only (ROM) and others as read-write (RAM). Writing to a ROM region would throw a SimulatorException with ErrorType.InvalidMemoryAddress. Real 8085 systems have ROM (containing the programme) starting at address 0x0000 and RAM starting at a higher address. This distinction is important for understanding embedded system memory maps.

Phase 6 Performance, Testing, and Ecosystem

Production-grade quality assurance and performance optimisation.

6.1 Multi-Threaded Execution Engine Medium Priority

Run the simulation engine on a background thread, with the UI on the main thread. This prevents the UI from freezing during long-running programmes. Java's ExecutorService or a dedicated Thread would run the fetch-decode-execute loop. Communication between the UI and engine threads would use a blocking queue or CompletableFuture. Thread safety for shared state (registers, memory) would be ensured via synchronized blocks or ReentrantLock.

6.2 Comprehensive JUnit Test Suite High Priority

Unit tests for every instruction. Each test would set up initial register/memory state, execute a single instruction, and verify the resulting register/memory/flag state against the expected values from the Intel 8085 User Manual. Edge cases include carry propagation, zero-flag behaviour on overflow, and auxiliary carry across nibble boundaries. The test suite would also include integration tests that run complete programmes and verify final state. JUnit 5 with parameterised tests (@ParameterizedTest) would enable testing all 8 register variants of each instruction with a single test method.

6.3 Demo Programmes Library High Priority

A built-in library of example programmes accessible from the menu: addition, subtraction, multiplication, division, sorting (bubble sort), factorial, Fibonacci, BCD arithmetic, string operations, subroutine calls, and interrupt handling. Each programme would include inline comments explaining the algorithm. The library would be loadable by number (e.g., "Enter demo number: 1" loads the addition programme). Storage: embed the programmes as string constants in a DemoPrograms class, or load from .asm files in a demos/ directory.

6.4 Code Optimiser / Peephole Analysis Low Priority

Analyse assembled code for common optimisation opportunities: replacing a MVI A, 00H with XRA A (saves one byte), replacing MOV A, A with NOP, detecting unreachable code after unconditional jumps. This would be a static analysis pass over the assembled machine code that suggests optimisations without applying them. Useful as a teaching tool for efficient 8085 programming.

6.5 Cross-Assembler to 8086 / Z80 Low Priority

Translate 8085 assembly into equivalent 8086 or Z80 assembly. The Z80 is the closest relative (it is upward-compatible with the 8080/8085 at the machine code level). The 8086 uses different mnemonics but many instructions have direct equivalents. A cross-assembler would be an educational tool showing how software portability worked between microprocessor generations.

6.6 Plugin Architecture for Custom I/O Devices Low Priority

Define a VirtualDevice interface with read(int port) and write(int port, int data) methods. Users could implement custom devices (virtual LED array, virtual keypad, virtual motor controller) and register them with the simulator. The IN and OUT instructions would dispatch to the registered device. This enables open-ended experimentation with I/O-driven programmes.

6.7 Instruction Set Reference / Built-In Help Medium Priority

A comprehensive built-in help system: typing HELP ADD would display the ADD instruction's syntax, operands, opcode encoding, affected flags, T-state count, and example usage. The reference data would be stored in a structured format (JSON or embedded constants) and queried by the help command. This turns the simulator into a self-contained learning environment.

6.8 Performance Benchmarking Mode Low Priority

Run a programme N times and report average execution time, instructions per second, and T-states per second. This is useful for benchmarking the simulator itself (how fast can our Java implementation simulate the 8085?) and for comparing different programme algorithms (which implementation of multiplication is more efficient in T-states?).

6.9 Automated Grading Integration Low Priority

For classroom use: the instructor provides a test harness (initial register/memory state and expected final state). Students submits their assembly programmes, and the simulator evaluates correctness automatically: "Your programme produced A=0x0C, expected A=0x0C: PASSED." This requires a batch mode API that accepts a programme, initial state, and expected state, and returns pass/fail with detailed diff.

6.10 Documentation Generator Low Priority

Generate HTML documentation from commented assembly source code, similar to Javadoc for Java. Comments following a defined format (e.g., beginning with ;;) would be extracted and formatted into a browsable HTML report with indices by label, cross-references, and call graphs.

6.11 Build System Migration (Maven/Gradle) Medium Priority

Currently, the project is compiled with a manual javac command. Migrating to Maven or Gradle would add dependency management, automated testing (run JUnit on mvn test), packaging (create a runnable .jar with mvn package), and CI/CD integration (GitHub Actions runs tests on every push). A pom.xml or build.gradle file would define the project structure, dependencies, and build lifecycle.

6.12 Syntax Highlighting Code Editor (CLI) Low Priority

Replace the raw line-by-line input with a terminal-based editor using a library like JLine 3 or Lanterna. This would provide command history (up/down arrows recall previous lines), tab completion for mnemonics, and ANSI colour highlighting for opcodes, registers, and comments. JLine integrates with the JVM's terminal handling to provide readline-like functionality.
